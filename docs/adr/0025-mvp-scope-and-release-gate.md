# ADR-0025：冻结 MVP 范围并以安全验收项作为发布门槛

- 状态：已接受
- 日期：2026-07-23

## 背景

前 25 轮已确定模板契约、行情口径、交互、运行限制和技术架构。继续逐项扩展会增加设计周期，也可能模糊“只操作最新时间段 A 至 D”这一核心安全边界，因此需要冻结首版范围并定义可测试的发布门槛。

## 决策

- MVP 只实现产品设计文档第 5.1 节列出的能力。
- 明确排除通用 Excel 编辑、多用户、原生客户端、批量文件、其他证券、日期替代、受保护列处理、多数据源静默降级和特殊 Excel 功能保真。
- 以 `acceptance-criteria.md` 的全部 P0 条目通过作为发布必要条件。
- `.xls` 和 `.xlsx` 必须分别进行模板兼容、动态写入、越界保护、保存重开和失败原子性测试。
- 实施分为技术验证、Excel Worker 与后端、PWA、集成测试和 Docker 部署，单人初步估算 13 至 20 个工作日。

## 影响

- 需求讨论在本决策后结束，产品设计 v1.0 成为 MVP 基线。
- 新增需求进入后续版本评估，不应在实现中隐式扩大首版范围。
- 如果技术验证不能证明 Apache POI 在兼容样本上满足写入边界，项目应停留在验证阶段并调整技术方案，而不是降低安全标准。

## 备选方案

- 继续逐项讨论后再冻结：会延长设计过程，当前剩余内容已可通过验收标准和技术验证闭环。
- 先开发后补验收：容易在 Excel 保真与越界写入问题上形成返工，未采用。
